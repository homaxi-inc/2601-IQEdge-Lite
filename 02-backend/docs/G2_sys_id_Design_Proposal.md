## 1. 你的主张（008 认同的方向）

| 观点 | 008 评审 |
|------|----------|
| `sys_id` = **内部全局唯一编号**，不含产品形态语义 | ✅ **正确** — 对标「资产 Tag / Fleet Unit ID」，不是「SKU 编码」 |
| `system_type`（iqwatch / iqbox / iqtrailer）在 **产测或部署** 才绑定 | ✅ **正确** — 属于 Registry **可变配置** |
| 同一物理资产从 IQWatch 迁到 IQBox → **re-assign**，不换号 | ✅ **正确** — 历史 Timestream、租约、审计连续 |
| 系统更健壮、灵活 | ✅ — 减少「前缀与现场不符」的永久技术债 |

**结论**: 方向与 G2「`sys_id` 对标 Tesla VIN、终身绑定物理资产」的 **本意一致**；当前 `IQW-/IQB-/IQT-` 前缀实现的是 **过度编码**，建议修正。

## 2. 现状问题（为何前缀方案脆弱）

```text
现状:  IQW-9041  →  语义上「永远是 Watch」
现场:  同一套核心部件改装箱体 →  要么造假前缀，要么新发号 →  历史断裂
```

| 问题 | 后果 |
|------|------|
| 前缀 ≡ `system_type` | Registry 与实物不一致时，API/报表/运维认知分裂 |
| Solar/AC 已共用 `IQB-` | 证明前缀本就不能表达全部形态，仍靠 `system_type` |
| 固件 NVS 烧录 `IQW-*` | 现场改型需返厂重烧或双轨 ID |
| 溢出「按产品线借千位块」 | 前缀方案下必要；中性 ID 下可 **单池简化** |

## 3. 推荐方案（008 · 方案 B — 中性 sys_id）

### 3.1 定义分离（铁律）

| 字段 | 含义 | 可变性 | 谁决定 |
|------|------|--------|--------|
| **`sys_id`** | 公司级 **Fleet 资产唯一 ID**（逻辑/物理资产 Tag） | **不可变**（终身） | 中央发号（产测首次写入） |
| **`system_type`** | 当前 **产品形态 / 能力模板** | **可变**（re-assign） | Registry（产测默认 or 部署确认） |
| **`component_id`** | 物理部件序列号（MPPT、Cerbo、RUT…） | 换件时变 | 现场 BOM / 维修 |
| **`deployment_state`** | 生命周期（仓库→现场→退役） | 按状态机变 | 运维 / 自动化 |

```text
         产测发号                    部署 / 改型
            │                            │
            ▼                            ▼
      sys_id = IQ-26-00001       system_type := iqbox
      (永不改)                    components[] 更新
```

### 3.2 推荐格式（待 §9 编号定稿）

> **讨论中** — 倾向 **`IQ-{YY}-{NNNNN}`**，非下表 `IQS-{serial}`。

```text
IQ-26-00001   ← 008 当前首选（见 §9）
```

| 项 | 草案（若改用 Y3 单池） |
|----|------------------------|
| 前缀 | **IQS** = IQ System |
| serial | 全局递增 |

**弃用**: `IQW-` / `IQB-` / `IQT-` 作为新设备 sys_id 前缀。

**为何不用 UUID**: 现场贴标、电话报修、Dealer 培训需要 **可读、可口述**；Tesla VIN 亦非 UUID。

**为何不用纯数字**: 前缀 `IQS-` 防与其他系统编号碰撞，且一眼识别 IQCloud Fleet。

### 3.3 `system_type` 何时写入（建议流程）

| 阶段 | `deployment_state` | `system_type` | 说明 |
|------|----------------------|---------------|------|
| 零件入库 | `inventory_parts` | `null` 或 `unassigned` | 仅 sys_id 标签 |
| 工厂组装 | `assembling` → `qa_bench` | **默认值**（如按工单 `iqwatch`） | 可改 |
| 出厂 | `in_transit` | 工单锁定值 | |
| 现场部署 | `deployed` | **部署工程师确认**（App/扫码） | 权威绑定 |
| 改型 / 重组 | `maintenance` | **PATCH re-assign** | **sys_id 不变** |

**API（将来）**:

```http
PATCH /api/v2/fleet/systems/{sys_id}/profile
{ "system_type": "solar_iqbox", "reason": "field_retrofit", "components": { ... } }
```

需 **审计日志**（谁、何时、从何形态改到何形态）— 满足租赁与合规。

### 3.4 固件 / MQTT（007 协作）

| 项 | 建议 |
|----|------|
| NVS / 产测 | 只烧录 **`sys_id`**（`IQS-*`），**不烧录** `system_type` |
| 上行 telemetry | **必填** `sys_id`；`system_type` **可选**（hint），**权威以 Registry 为准** |
| 改型 | 无需返厂重烧号；云端 PATCH + 可选 Shadow 下发新能力模板 |

与 [`FIRMWARE_ALIGNMENT_007.md`](../../09-contract/schemas/energy/FIRMWARE_ALIGNMENT_007.md) 差异：删除「格式 IQW/IQB/IQT」；保留 `sys_id` 必填。

### 3.5 存储与 API（无结构性破坏）

| 层 | 影响 |
|----|------|
| Timestream dimension | 仍为 `sys_id` — **历史连续** |
| Fleet API 路径 | `/systems/{sys_id}/...` **不变** |
| 列表过滤 | `?system_type=iqwatch` **仍有效**（查 Registry 字段） |
| Legacy 合流 | `aliases.mppt_serial` **不变** |

## 9. sys_id 编号规则 — 与 Bob 共定（2026-05-29 讨论）

### 9.1 您的示例：`iq-26-06001`

拆解（008 理解）：

```text
iq  ─  26  ─  06001
│      │       └── 顺序号（5 位，左补零）
│      └── 年份：2026 → 26
└── 品牌/系统前缀（IQ）
```

| 优点 | 说明 |
|------|------|
| **中性** | 不含 Watch/Box/Trailer |
| **含年份** | 报表可按年统计出货量；贴标一眼知批次 |
| **顺序号** | 产线、仓库、客服都好念 |
| **与旧 6001 心理连续** | `06001` 可接 Trailer 原「6001 段」叙事（若您希望） |

| 需统一 | 008 建议 |
|--------|----------|
| 大小写 | **规范存储 `IQ` 大写**；API/扫码可容错 `iq` → 归一化 |
| 分隔符 | **保留两杠** `IQ-26-06001`（可读性优于 `IQ2606001`） |
| 年份 2 位 vs 4 位 | 见方案对比 §9.3 |
| 「26」含义 | **发号年**（产测首次分配），**终身不变** — 不是部署年、不是改型年 |

### 9.2 三种编号方案（请选一或混搭）

#### 方案 **Y1** — `IQ-{YY}-{NNNNN}`（008 首选 · 贴近您的例子）

```text
IQ-26-00001
IQ-26-00002
…
IQ-26-99999   （单年上限 ~10 万台）
IQ-27-00001   （新年份重置序号）
```

| 段 | 规则 |
|----|------|
| `IQ` | 固定前缀（IQEdge Fleet） |
| `YY` | **发号公历年后两位**（2026→`26`） |
| `NNNNN` | **5 位数字**，该年内 **原子递增**，从 `00001` 起 |

**JSON Schema 正则（草案）**:

```text
^IQ-[0-9]{2}-[0-9]{5}$
```

**容量**: 每年 99,999 台；对特种 IoT 足够。跨百年时 `00`–`99` 可循环（2100 年与 2200 年运维流程再评估即可）。

---

#### 方案 **Y2** — `IQ-{YYYY}-{NNNNN}`（年份更直白）

```text
IQ-2026-00001
```

| 对比 Y1 | |
|---------|--|
| ✅ 国际团队零歧义 | ❌ 多 2 字符；贴标略长 |
| ✅ 避免「26 = 型号？」误解 | |

适合：出口、多工厂、ERP 对接多的一方。

---

#### 方案 **Y3** — `IQS-{NNNNNN}`（无年份 · 原 §3.2）

```text
IQS-000001 … IQS-999999
```

| 对比 Y1 | |
|---------|--|
| ✅ 最短、发号器最简单 | ❌ 看不出批次年；要靠 Registry `issued_at` |
| ✅ 序号永不「跨年重置」 | |

适合：极重 **单池全局唯一**、不愿维护「每年计数器」的团队。

---

### 9.3 方案对照表

| | **Y1 `IQ-26-06001`** | **Y2 `IQ-2026-06001`** | **Y3 `IQS-09041`** |
|--|----------------------|-------------------------|---------------------|
| 中性产品形态 | ✅ | ✅ | ✅ |
| 年份可见 | ✅ 2 位 | ✅ 4 位 | ❌（靠元数据） |
| 贴标长度 | 11 字符 | 13 字符 | 9 字符 |
| 发号器 | 每年一个计数器 | 同左 | 全局一个计数器 |
| 与您示例 | **最近** | 略改 | 不同 |

**008 推荐**: **Y1** — 与您的 `iq-26-06001` 一致，仅规范为 **`IQ-26-06001`（大写 + 首年可从 `00001` 起）**。

### 9.4 发号细则（Y1 定稿后写入 System Model §4）

| 规则 | 建议 |
|------|------|
| **发号时点** | 产测首次入库 / 首次写入 Registry 时分配；**终身不变** |
| **年份取法** | 服务器 UTC **公历年** 的 `YY`；12 月 31 日 23:59 仍为当年池 |
| **序号起点** | **默认 `00001`/`年`**（推荐，最清晰） |
| **序号起点（备选）** | 2026 年从 **`06001`** 起，对接旧 Trailer「6001 段」叙事 — **仅当您明确要求** |
| **跨年** | `IQ-27-00001` 自动新开池；**不**占用前一年未用完号码 |
| **全局唯一** | 全公司 **dev/prod 共用** 同一套 `sys_id`（与原 §4.4 一致） |
| **Reserved** | `IQ-26-00000`、全 `9` 测试号等不入生产库 |
| **展示** | UI 可显示 `IQ-26-6001`（去前导零）但 **存储/ API 用零填充 5 位** |
| **归一化** | ingest/API：`iq-26-6001` → `IQ-26-06001` |

```text
Provisioning（伪逻辑）:
  yy  = current_year % 100
  n   = atomic_increment(f"sys_id_counter_{yy}")   # DynamoDB / DB
  sys_id = f"IQ-{yy:02d}-{n:05d}"
```

## 6. 待 Bob 拍板的问题

### 编号（§9.6 摘要）

| ID | 问题 | 008 建议 |

### Q1 — 已并入 N1/N2（原 IQS 前缀问题作废，改用 `IQ-YY-NNNNN`）

### Q2 — 已写入文档/报告的 `IQW-9041` 如何处理？

- **A**: **保留为 sys_id**（ grandfather ），新设备用 `IQS-*`；Registry `id_format: legacy_prefixed | neutral`
- **B**: **一次性映射** `IQW-9041` → `IQS-09041`，旧 ID 进 `aliases.legacy_sys_id`
- **C**: 仅新批次 `IQ-26-*`，旧 `IQW-*` 永不改（两套并存）

**008 倾向**: **C** — HIL 已用 `IQW-9041` 不动；2026 起统一 `IQ-YY-NNNNN`。

### Q3 — `system_type` 未分配时 API 行为？

- **A**: 允许 `system_type: null`，energy API 按 **实际 components** 推断
- **B**: 禁止 `deployed`  without `system_type`（部署流程强制扫码确认）

**008 倾向**: **B** 对 `deployed`；产测阶段允许 null。

### Q4 — re-assign 是否允许跨 `system_type` 大类？

例：`iqwatch` → `iqtrailer`（差异大：Cerbo、多相机）

- **A**: 允许，但需 **Admin 权限** + 审计 + 组件清单校验
- **B**: 仅允许「近邻」改型（watch ↔ solar box），禁止 watch → trailer

**008 倾向**: **A + 强校验** — 业务上少见，但应用规则约束而非 ID 前缀禁止。
